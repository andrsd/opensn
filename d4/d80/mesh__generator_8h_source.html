14/**
15 * Mesh generation can be very complicated in parallel. Some mesh formats
16 * do not store connectivity information and therefore we have to establish
17 * connectivity after the file is read. Most mesh formats are also not
18 * partitioned for multiple processors when read/generated. The design of these
19 * types of objects need to consider the previous generation of how we did this.
20 * We had the concept of a VolumeMesher, which essentially took an
21 * `UnpartitionedMesh` and converted it into a partitioned mesh (i.e. the
22 * required `MeshContinuum` object). Up to now we've really only had two
23 * variants. The `VolumeMesherPredefinedUnpartitioned` and the
24 * `VolumeMesherExtruder`. Both of which operated on an `UnpartitionedMesh`
25 * object. With this new design we want to unify these concepts to make them
26 * more extendable and therefore we split a `MeshGenerator`'s execution into a
27 * phase that generates an unpartitioned mesh and a phase that then converts
28 * this mesh into real mesh (with both steps customizable). The phase that
29 * creates the real mesh can be hooked up to a partitioner that can also be
30 * designed to be pluggable.
31 */
